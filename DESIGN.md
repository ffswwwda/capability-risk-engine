## 1. 产品哲学：一套思考路径，适配所有岗位

核心定位（2026-07-20 确立）：

> 做一个**通用的"思考引擎"**，任何岗位 / 任何人都通过同一套思考路径工作，
> 差异只在"知识库"（岗位领域数据）与"经验"不同——**思考 / 输出的路径相同**。

架构本质 = 经典「推理机 + 知识库」分离：

- **引擎内核**（唯一路径）：毛选方法论（矛盾论 / 调查研究 / 实践论 / 实事求是 + 四步循环）。
- **岗位知识库**（可插拔）：JSON 加载，描述该岗位的能力维度、环境类型、矛盾库、结果路径。
- 用户原话："掌握岗位知识库 + 接入 AI = 替代该岗位"。

工程落点（当前文件）：

- `capability-risk-engine/index.html` 纯规则 MVP。
- `analyze(text, kb)` = 引擎内核，岗位无关，只认 KB Schema。
- `KB_LIBRARY` = 内置知识库样例（研究员 / 运营 / 产品经理）。
- `normalizeKB(o)` = 接收外部 JSON，规范化为引擎可识别的 KB Schema。
- 加新岗位 = 写一份符合 Schema 的 KB JSON，引擎自动适配，无需改内核。

## 2. 毛选方法论技能栈（引擎内核来源）

### 2.1 为什么是毛选

引擎内核不是拍脑袋设计的流程，而是把毛泽东方法论著作里的**可操作分析技能**抽出来，
变成可程序化的"判断步骤"。原因：

- 毛选方法论天然是**反教条、反本本主义**的——它强调"先看眼前矛盾是什么质地，再决定怎么做"，
  这与"通用引擎 + 可插拔知识库"的架构同构：引擎不预设答案，只提供路径。
- 它的几个核心技能（实事求是 / 调查研究 / 矛盾特殊性 / 重点论 / 实践闭环 / 群众路线）
  恰好对应一个"接到任务 → 看清情况 → 抓关键 → 下判断 → 验证"的完整闭环。

### 2.2 六层技能栈详解

下面每一项都标注：原文出处、核心命题、在引擎里的工程落点。

#### ① 实事求是（认识论总纲）
- **出处**：《改造我们的学习》（1941）。
- **核心命题**："实事"就是客观存在着的一切事物，"是"就是客观事物的内部联系（规律），
  "求"就是我们去研究。一切从客观真实出发，不凭主观想象。
- **在引擎里**：所有量化结论必须基于"真实命中数据"，不臆测。
  例如能力覆盖分只由触发词命中数决定（`cov = 命中 ? min(5,3+命中数) : 2`），
  匹配度 = 能力均值 / 难度 × 100，全部可算、可溯源。
- **对应引擎标注**：`实事求是·基于真实数据`（step0 / step3 / step6）。

#### ② 调查研究（前提）
- **出处**：《反对本本主义》（1930）"没有调查，没有发言权"；《〈农村调查〉的序言和跋》。
- **核心命题**：结论产生于调查情况的末尾，而不是在它的先头；只有先采集真实情况，才能谈判断。
- **在引擎里**：接到任务后第一步是"采"——把输入拆句、逐类环境做触发词核对，
  不急于下结论。未识别到矛盾时，引擎明确走"情况A：先补调查，不急于下结论"分支。
- **对应引擎标注**：`调查研究·先采真实情况`（step1 / step2）。

#### ③ 实践论（认识—实践—再认识闭环）
- **出处**：《实践论》（1937）。
- **核心命题**：认识从实践始，经过实践得到了理论的认识，还须再回到实践去检验、修正。
- **在引擎里**：结论不是终点。`step8 反馈闭环` 预留接口——真实结果回流后，
  反向修正矛盾权重与能力基线。当前 MVP 暂未接入，但接口已留。
- **对应引擎标注**：`实践论·认识-实践闭环`（step8；矛盾库 c4 系列）。

#### ④ 具体问题具体分析（矛盾特殊性）
- **出处**：《矛盾论》（1937）"矛盾的特殊性"一节，引列宁："马克思主义的活的灵魂：
  具体地分析具体的情况"；毛泽东发展出"不同质的矛盾，只有用不同质的方法才能解决"。
- **核心命题**：拒绝套模板。看清眼前这个矛盾**本身是什么质地**，再选方法。
  这是毛泽东与王明"本本主义"的根本分水岭。
- **在引擎里**：它是**贯穿调度层的元原则**（详见第 5 节），最集中的工程表达是
  路由追踪页那句话——"引擎不生成结论，它只是按你定义的规则，把输入片段映射到结论。
  规则透明、可改、可溯源"。
- **对应引擎标注**：它不绑定某一步，而是调度层整体的灵魂；矛盾路由（step4）是其最直接体现。

#### ⑤ 抓主要矛盾（重点论 / 两点论与重点论）
- **出处**：《矛盾论》"矛盾的特殊性"与"主要的矛盾和主要的矛盾方面"两节。
- **核心命题**：在复杂事物中，诸多矛盾地位不平衡，必有一个是主要的、起决定作用的；
  要抓住它，同时不忽略次要矛盾的转化。
- **在引擎里**：`step4 主矛盾路由` 按触发词命中数排序，取前 2 条作为主矛盾；
  `step7 红绿灯` "有红则红"——只要关键风险红灯，整体即停。
- **对应引擎标注**：`矛盾论·抓主要矛盾`（step4 / step7）。

#### ⑥ 群众路线（价值闭环）
- **出处**：《关于领导方法的若干问题》（1943）"从群众中来，到群众中去"。
- **核心命题**：将群众的意见集中起来，化为系统的意见，又到群众中坚持下去。
  价值闭环，不是自说自话。
- **在引擎里**：知识库由岗位人自己定义（可插拔 = "从岗位中来"）；
  输出是给岗位人**可执行的决策建议 + 闭环接口**（"到岗位中去"）。
  9 宫格看板整体导向"决策"，而非炫技。
- **对应引擎标注**：体现在整体输出导向与 KB 可插拔机制，非单步标签。

### 2.3 技能栈分层关系（文字图）

```
        ┌─────────────────────────────┐
        │   ⑥ 群众路线（价值闭环）      │  ← 输出导向：可执行的、可插拔的
        └───────────────┬─────────────┘
                        │
        ┌───────────────┴─────────────┐
        │   矛盾分析法层（调度核心）      │
        │  ④ 具体问题具体分析(特殊性)    │  ← 元原则：先看矛盾是什么质地
        │  ⑤ 抓主要矛盾(重点论)         │  ← 取前2、有红则红
        └───────────────┬─────────────┘
                        │
   ┌────────────────────┼────────────────────┐
   │ ① 实事求是(总纲)    │ ② 调查研究(前提)    │ ③ 实践论(闭环)
   │ 量化只基于真实命中   │ 先采真实情况再判断  │ 结论回流再修正
   └────────────────────┴────────────────────┘
```

## 3. 9 宫格调度层 = 四步循环的工程实现

### 3.1 四步循环 vs 9 宫格对照

引擎对外呈现为「9 宫格决策看板」，对内是「四步循环」逐步执行。二者是同一过程的两面：

| 四步循环（内核逻辑） | 9 宫格（看板呈现） | 引擎 step |
|---|---|---|
| 知：加载规则库 | （知识层，不占 9 格） | step0 |
| ① 切割目标与风险 | ① 结果锚点 | step1 |
| ② 看清环境状况 | ② 环境状况 | step2 |
| ③ 量化自身能力 | ③ 能力范围 | step3 |
| ④ 抓主要矛盾 | ④ 结果路径图 / ⑤ 判断原则 | step4 / step5 |
| ⑤ 分支判断 | ⑤ 判断原则 | step5 |
| ⑥ 算匹配度 | ⑥ 匹配矩阵 | step6 |
| ⑦ 给决策灯 | ⑧ 决策红绿灯 | step7 |
| ⑧ 留反馈闭环 | ⑨ 反馈闭环 | step8 |
| （能力雷达） | ⑦ 能力雷达 | step7 内 |
| （路径图） | ④ 结果路径图 | step4 内 |

> 注：9 格中的 ④⑦⑧⑨ 与 ①②③⑤⑥ 在循环里分属不同 step，看板为视觉平衡做了排布，
> 逻辑上仍严格对应四步循环。

### 3.2 每一步的毛选归属（详表）

| step | 名称 | 毛选归属 | 工程动作 |
|---|---|---|---|
| 0 | 加载知识库 | 实事求是·基于真实数据 | 读取 KB 规则（能力/环境/矛盾/路径） |
| 1 | ① 结果锚点 | 调查研究·先采真实情况 | 输入拆句，逐句判目标/风险归属 |
| 2 | ② 环境状况 | 调查研究·先采真实情况 | 逐类环境触发词核对 |
| 3 | ③ 能力范围 | 实事求是·基于真实数据 | 逐维能力触发词核对，给覆盖分 |
| 4 | 主矛盾路由 | 矛盾论·抓主要矛盾 | 逐条矛盾触发词核对，排序取前2 |
| 5 | ⑤ 判断原则 | 矛盾论 + 实践论 | 分支：有矛盾? 能力≥阈值? |
| 6 | ⑥ 匹配矩阵 | 实事求是·基于真实数据 | 匹配度 = 能力均值/难度×100，定灯 |
| 7 | ⑧ 决策红绿灯 | 矛盾论·抓主要矛盾 | 红>0则红，否则黄>0则黄，否则绿 |
| 8 | ⑨ 反馈闭环 | 实践论·认识-实践闭环 | 预留真实结果回灌接口 |

### 3.3 核心函数映射（`analyze()` 逐 step）

```
analyze(text, kb)
 ├─ mk(知识, 0, 加载知识库,      实事求是)      → out.steps[0]
 ├─ mk(加工, 1, ①结果锚点,       调查研究)      → out.goals / out.risks
 ├─ mk(加工, 2, ②环境状况,       调查研究)      → out.env
 ├─ mk(加工, 3, ③能力范围,       实事求是)      → out.caps / capCov
 ├─ mk(加工, 4, 主矛盾路由,      矛盾论·抓主要矛盾) → out.principal
 ├─ mk(加工, 5, ⑤判断原则,       矛盾论+实践论)  → out.judge / branch
 ├─ mk(加工, 6, ⑥匹配矩阵,       实事求是)      → out.matrix
 ├─ mk(加工, 7, ⑧决策红绿灯,     矛盾论·抓主要矛盾) → out.light
 └─ mk(输出, 8, ⑨反馈闭环,       实践论)        → out.loop（接口预留）
```

| **⑤ 抓主要矛盾** | ④主矛盾路由 + ⑧红绿灯 | 4 / 7 | `out.principal`(前2), `out.light` | 矛盾按命中数排序取前2；有红则红 | 在诸多矛盾中抓决定性2条，关键风险一票否决 |
| **⑥ 群众路线** | 整体输出导向 + KB 可插拔 | 全周期 | `KB_LIBRARY`, `normalizeKB` | 知识库由岗位人写（从岗位来）；输出给岗位人可执行建议（到岗位去） | 从岗位中来、到岗位中去，价值闭环 |

跨技能联动示例（以研究员 KB 的一条矛盾为例）：

- 输入含"样本小" → `调查研究`②环境/①锚点采集到片段；
- 命中 `c1 数据不足 vs 决策紧迫`（矛盾特殊性 = 这条矛盾是此岗位当前的具体矛盾）→ `具体问题具体分析`④路由选中；
- 按命中数排进 `抓主要矛盾`⑤前2 → `实事求是`③算能力均值；
- 若均值 < 阈值 → `判断原则`走"难覆盖"分支 → `红绿灯`⑦给红灯；
- 推荐方法"先补 VOC 采集"回流 → 未来 `实践论`③闭环修正基线。

## 5. "具体情况具体分析"在调度层的精确落点（重点展开）

### 5.1 它是元原则，不是某一步

"具体情况具体分析"不绑定单个 step，而是**调度层整体的灵魂**。
其他五项技能都能找到对应 step，唯独它体现在"引擎为什么这样跑"的根本设定上：

> 引擎内核不认识任何具体岗位，它只认 KB Schema。
> 同一个引擎，换一份知识库，跑出的矛盾、方法、决策完全不同——
> 因为"矛盾是什么质地"由该岗位的知识库定义，而非引擎硬编码。

这就是"不同质的矛盾，只有用不同质的方法才能解决"的程序化表达。

### 5.2 三个工程表达

**(a) KB 可插拔 = 承认矛盾特殊性**
每个岗位的矛盾库不同（研究员的"数据不足 vs 决策紧迫"、运营的"预算有限 vs 目标过高"、
PM 的"需求伪繁荣 vs 真实价值"），引擎不预设"通用答案"，只提供"通用路径"。
这正是对"本本主义"（拿一套模板套所有情况）的拒绝。

**(b) 矛盾路由 = 对当前输入做特殊性判定**
`step4` 不是套固定结论，而是：把当前具体输入，按具体触发词，命中具体矛盾，
按命中数排序取前 2。输入改几个词，主矛盾和决策就变——这就是"具体分析"的运行时。

**(c) 全透明溯源 = 让岗位人自己校验特殊性判定**
路由追踪页（`out.trace`）把每条结论反向追到：原始输入片段 → 触发规则 → 命中关键词。
岗位人能看到"为什么是这个结论"，并直接改规则。可改、可溯源，是"具体分析"可信的前提。

### 5.3 与"本本主义"的对立

| 本本主义（反模式） | 具体情况具体分析（本引擎） |
|---|---|
| 不管什么任务都走同一条 pipeline | 先判定矛盾特殊性，再调度加工路径 |
| 结论由模板生成 | 结论由"输入片段 → 规则映射"生成 |
| 规则黑箱、不可改 | 规则透明、可改、可溯源 |
| 引擎硬编码岗位知识 | 知识库可插拔，引擎与岗位解耦 |

## 6. 知识库 Schema 与可插拔机制

### 6.1 Schema 字段说明

```jsonc
{
  "role": "岗位名",
  "intro": "一句话说明该库视角",
  "judge": { "capThreshold": 3.2 },          // 能力均值达标阈值
  "capDims": [                                // 能力维度（实事求是·量化对象）
    { "name": "维度名", "key": "触发词1|触发词2", "baseline": 3 }
  ],
  "envTypes": [                               // 环境类型（调查研究·采集对象）
    { "name": "环境名", "kw": ["关键词"] }
  ],
  "pathSteps": ["步骤1", "步骤2"],            // 结果路径图
  "contradictions": [                         // 矛盾库（矛盾特殊性·本岗位专属）
    { "id": "c1", "name": "矛盾A vs 矛盾B",
      "kw": ["词"], "method": "推荐方法", "source": "矛盾论·抓主要矛盾" }
  ]
}
```

### 6.2 内置三库的毛选触发词分布

| 知识库 | 矛盾条数 | source 分布 | 典型矛盾（体现特殊性） |
|---|---|---|---|
| 研究员 | 4 | 矛盾论×3 / 实践论×1 | 数据不足 vs 决策紧迫；表面需求 vs 真实动机 |
| 运营 | 4 | 矛盾论×2 / 实践论×1 / 实事求是×1 / 调查研究×1 | 预算有限 vs 目标过高；平台依赖 vs 自有阵地 |
| 产品经理 | 4 | 矛盾论×2 / 实事求是×2 | 需求伪繁荣 vs 真实价值；范围蔓延 vs 按期交付 |

### 6.3 如何加一个新岗位（增加阶段示例）

1. 复制任一内置 KB 结构，改 `role` / `capDims` / `envTypes` / `contradictions`。
2. 矛盾库按"该岗位真实会遇到的矛盾"写，每条标 `source`（毛选归属）。
3. 通过「工作台 → 从文件加载 / 粘贴 JSON」加载，`normalizeKB` 自动校验并接入。
4. 引擎内核零改动，9 宫格与运行路径自动适配。
